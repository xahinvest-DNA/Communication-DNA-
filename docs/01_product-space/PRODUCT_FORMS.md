# PRODUCT FORMS

## Goal of this document
Map possible product forms without choosing one final form too early.

## Candidate forms

### 1. Mobile application
Strengths:
- can hold history, patterns, and profile;
- supports repeated use;
- allows multi-step flows;
- stronger for long-term behavioral learning.

Risks:
- higher implementation cost;
- more demanding onboarding;
- privacy concerns become central.

### 2. Telegram bot / chat bot
Strengths:
- fast entry;
- low friction for real-life testing;
- easy “forward message -> get analysis” use case.

Risks:
- can collapse into generic advice chat;
- weak long-term visualization;
- harder to express structural analysis.

### 3. Private desktop / local-first tool
Strengths:
- privacy trust;
- can process larger histories;
- better for deep pattern analysis.

Risks:
- lower mass adoption;
- weaker everyday convenience.

### 4. Share-to-app utility layer
Strengths:
- works in the moment before reply;
- useful as a pause-and-check mechanism;
- integrates into actual communication behavior.

### 5. Hybrid model
A simple front layer for immediate use, plus a deeper profile/history layer behind it.
This may be the strongest long-term direction.

## Product-form evaluation criteria
Every form should later be evaluated by:
- speed to first value;
- privacy trust;
- ability to preserve context;
- ability to teach the user over time;
- suitability for emotionally loaded scenarios;
- capacity to support multi-step outputs rather than one-shot answers.

## Working note
At this stage, no single form is locked. The theory should stay portable across forms.
